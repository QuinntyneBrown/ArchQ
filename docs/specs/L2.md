# L2 - Detailed Requirements

---

## L2-001: Tenant Provisioning
**Traces to:** L1-001

The system must allow creation of a new tenant (organization) with a unique identifier, display name, and slug. Each tenant must operate in complete isolation — no tenant may access another tenant's ADRs, users, or configuration.

**Acceptance Criteria:**
1. Given an administrator creates a new organization with name "Acme Corp" and slug "acme-corp", when the organization is saved, then it must be assigned a unique tenant ID and be accessible via its slug.
2. Given two tenants "Acme Corp" and "Beta Inc" exist, when a user belonging to "Acme Corp" queries ADRs, then zero ADRs from "Beta Inc" must be returned regardless of query parameters.
3. Given a tenant slug "acme-corp" already exists, when another organization attempts to use the same slug, then the system must reject the request with a conflict error.

---

## L2-002: Tenant-Scoped Data Isolation
**Traces to:** L1-001

Every database query and API request must be scoped to the authenticated user's active tenant. The system must enforce tenant scoping at the data access layer so that application-level bugs cannot leak data across tenants.

**Acceptance Criteria:**
1. Given a user authenticated under tenant A, when the user crafts an API request referencing a resource ID belonging to tenant B, then the system must return a 404 (not 403) to avoid confirming the resource's existence.
2. Given a direct database query is executed with a missing tenant filter, when the query reaches the data access layer, then the system must reject or automatically scope the query to the current tenant context.

---

## L2-003: User Registration and Authentication
**Traces to:** L1-002

The system must allow users to register with an email address and password, verify their email, and authenticate via secure credential exchange. Passwords must be hashed using bcrypt (or equivalent) with a minimum cost factor of 12.

**Acceptance Criteria:**
1. Given a new user registers with email "user@example.com" and a valid password, when registration completes, then the system must send an email verification link and the account must remain inactive until verified.
2. Given a registered and verified user, when they submit valid credentials, then the system must return a signed JWT access token (expiry <= 15 minutes) and a refresh token (expiry <= 7 days).
3. Given a user submits an incorrect password, when authentication is attempted, then the system must return a generic "invalid credentials" error without revealing whether the email exists.
4. Given a user has failed authentication 5 times within 10 minutes, when they attempt again, then the system must temporarily lock the account for 15 minutes and notify the user via email.

---

## L2-004: Organization Membership
**Traces to:** L1-002

A user must be associated with at least one organization. The system must support users belonging to multiple organizations and allow switching between them within a single session.

**Acceptance Criteria:**
1. Given a user belongs to organizations "Acme Corp" and "Beta Inc", when the user views the organization switcher, then both organizations must be listed.
2. Given a user switches from "Acme Corp" to "Beta Inc", when the switch completes, then all subsequent requests must be scoped to "Beta Inc" and the UI must reflect the active organization.
3. Given a user belongs to only one organization, when they log in, then the system must automatically set that organization as the active context without prompting.

---

## L2-005: Role Definitions
**Traces to:** L1-003

The system must support the following roles within each tenant: **Admin** (full tenant management), **Author** (create and edit own ADRs), **Reviewer** (review and approve/reject ADRs), and **Viewer** (read-only access). A user may hold multiple roles within a single tenant.

**Acceptance Criteria:**
1. Given a user with the "Author" role, when they attempt to approve an ADR, then the system must deny the action with a 403 Forbidden response.
2. Given a user with the "Admin" role, when they assign the "Reviewer" role to another user, then that user must immediately gain reviewer capabilities without re-authentication.
3. Given a user with the "Viewer" role, when they attempt to create an ADR, then the system must deny the action with a 403 Forbidden response.
4. Given a user with both "Author" and "Reviewer" roles, when they create an ADR, then they must not be able to approve their own ADR.

---

## L2-006: Role Assignment and Management
**Traces to:** L1-003

Tenant admins must be able to assign and revoke roles for users within their organization. Role changes must take effect immediately and must be recorded in the audit trail.

**Acceptance Criteria:**
1. Given an admin revokes the "Author" role from a user who has ADRs in Draft status, when the role is revoked, then those Draft ADRs must remain in the system and the user must retain read access to their own ADRs but lose edit capability.
2. Given an admin assigns the "Reviewer" role to a user, when the assignment is saved, then the audit trail must record the admin who made the change, the target user, the role granted, and the timestamp.

---

## L2-007: ADR Creation with Markdown Editor
**Traces to:** L1-004

The system must provide an integrated Markdown editor for authoring ADR content. The editor must support standard Markdown syntax (headings, lists, code blocks, tables, links, images) with a live preview pane.

**Acceptance Criteria:**
1. Given an Author navigates to "Create ADR", when the page loads, then a Markdown editor must be displayed with a split-pane layout: edit area on the left and live preview on the right.
2. Given an Author types `## Decision` in the editor, when the preview updates, then it must render an H2 heading with the text "Decision".
3. Given an Author pastes a fenced code block with a language identifier, when the preview renders, then the code must be syntax-highlighted.
4. Given an Author creates a new ADR, when they save it, then the ADR must be assigned a sequential number within the tenant (e.g., ADR-001, ADR-002) and its initial status must be "Draft".

---

## L2-008: ADR Editing
**Traces to:** L1-004

Authors must be able to edit ADRs they created while in Draft or In Review status. Admins must be able to edit any ADR in Draft or In Review status. The system must prevent editing of ADRs in terminal statuses (Approved, Rejected, Superseded, Deprecated).

**Acceptance Criteria:**
1. Given an ADR in "Draft" status, when the original Author opens it, then the Markdown editor must be enabled with the current content loaded.
2. Given an ADR in "Approved" status, when any user opens it, then the editor must be read-only and the system must display a message indicating the ADR is finalized.
3. Given an ADR in "In Review" status, when the Author edits and saves content, then the system must create a new revision, increment the version number, and notify all assigned approvers of the change.

---

## L2-009: ADR Markdown Template
**Traces to:** L1-004

The system must provide a default ADR Markdown template containing sections: Title, Status, Context, Decision, Consequences, and Participants. Tenant admins must be able to customize the default template for their organization.

**Acceptance Criteria:**
1. Given an Author creates a new ADR, when the editor opens, then it must be pre-populated with the tenant's configured template (or the system default if none is configured).
2. Given a tenant admin updates the default template, when an Author subsequently creates a new ADR, then the updated template must be used.
3. Given a tenant admin saves a template with invalid Markdown, when the template is validated, then the system must accept it (Markdown is freeform) but must warn if required section headings (Title, Status, Context, Decision, Consequences) are missing.

---

## L2-010: ADR Workflow Status Transitions
**Traces to:** L1-005

ADRs must follow the defined state machine: Draft -> In Review -> Approved | Rejected. Approved ADRs may transition to Superseded or Deprecated. Rejected ADRs may return to Draft. Each transition must be validated against permitted transitions.

**Acceptance Criteria:**
1. Given an ADR in "Draft" status, when the Author submits it for review, then its status must change to "In Review" and the Author must assign at least one approver.
2. Given an ADR in "In Review" status, when all required approvers have approved, then its status must automatically transition to "Approved".
3. Given an ADR in "In Review" status, when any approver rejects it, then its status must transition to "Rejected" with the rejection reason recorded.
4. Given an ADR in "Rejected" status, when the Author chooses to revise it, then its status must return to "Draft" and all previous approval decisions must be cleared.
5. Given an ADR in "Approved" status, when an Author submits a new ADR that supersedes it, then the original ADR's status must transition to "Superseded" with a reference to the new ADR.
6. Given an ADR in "Draft" status, when a user attempts to transition it directly to "Approved", then the system must reject the transition as invalid.

---

## L2-011: Workflow Transition Notifications
**Traces to:** L1-005

The system must notify relevant users when an ADR transitions between statuses. Notifications must be delivered in-app and optionally via email.

**Acceptance Criteria:**
1. Given an ADR moves from "Draft" to "In Review", when the transition completes, then all assigned approvers must receive an in-app notification and an email (if email notifications are enabled for their account).
2. Given an ADR moves from "In Review" to "Approved", when the transition completes, then the Author must receive a notification.
3. Given an ADR moves from "In Review" to "Rejected", when the transition completes, then the Author must receive a notification that includes the rejection reason.

---

## L2-012: Approver Assignment
**Traces to:** L1-006

When submitting an ADR for review, the Author must assign one or more users with the "Reviewer" role as approvers. The system must validate that assigned approvers hold the Reviewer role and are not the ADR's author.

**Acceptance Criteria:**
1. Given an Author submits ADR-005 for review and assigns users R1 and R2 as approvers, when the submission is saved, then R1 and R2 must appear as pending approvers on ADR-005.
2. Given an Author attempts to assign themselves as an approver, when the assignment is validated, then the system must reject it with an error: "Authors must not approve their own ADRs."
3. Given an Author attempts to assign a user with only the "Viewer" role as an approver, when the assignment is validated, then the system must reject it with an error indicating the user lacks the Reviewer role.

---

## L2-013: Approval and Rejection Decisions
**Traces to:** L1-006

Each assigned approver must be able to record an "Approve" or "Reject" decision with an optional comment. The system must track the decision, the approver's identity, the timestamp, and any comment.

**Acceptance Criteria:**
1. Given approver R1 opens ADR-005 in "In Review" status, when R1 clicks "Approve" and adds the comment "LGTM", then the system must record R1's approval with the comment and timestamp.
2. Given approver R2 opens ADR-005 in "In Review" status, when R2 clicks "Reject" and provides a reason "Insufficient cost analysis", then the system must record R2's rejection with the reason and timestamp.
3. Given approver R1 has already approved ADR-005, when R1 attempts to approve again, then the system must prevent the duplicate decision.
4. Given an ADR requires 2 approvals and has received 1 approval and 0 rejections, when the status is evaluated, then the ADR must remain "In Review".

---

## L2-014: Approval Threshold Configuration
**Traces to:** L1-006

Tenant admins must be able to configure the minimum number of approvals required for an ADR to advance from "In Review" to "Approved". The default threshold must be 1.

**Acceptance Criteria:**
1. Given a tenant admin sets the approval threshold to 3, when an ADR in review receives 2 approvals and 0 rejections, then the ADR must remain in "In Review" status.
2. Given a tenant admin sets the approval threshold to 3, when an ADR receives its 3rd approval, then the ADR must automatically transition to "Approved".
3. Given the approval threshold is not configured for a tenant, when an ADR receives 1 approval, then the ADR must transition to "Approved" (default threshold of 1).

---

## L2-015: Meeting Notes
**Traces to:** L1-007

Users must be able to attach meeting notes to ADRs in "Draft" or "In Review" status. Meeting notes must include a title, date, attendees list, and Markdown body content.

**Acceptance Criteria:**
1. Given an ADR in "Draft" status, when an Author adds a meeting note with title "Architecture Review - 2026-04-10", date "2026-04-10", attendees ["Alice", "Bob"], and body content, then the meeting note must be saved and associated with the ADR.
2. Given an ADR in "Approved" status, when a user attempts to add a meeting note, then the system must reject the action with an error indicating that artifacts must not be added to finalized ADRs.
3. Given an ADR with 3 meeting notes, when a user views the ADR detail page, then all 3 meeting notes must be listed in reverse chronological order with title, date, and attendee count visible.

---

## L2-016: General Notes
**Traces to:** L1-007

Users must be able to attach general notes to ADRs in "Draft" or "In Review" status. Notes must include a title and Markdown body content authored via the Markdown editor.

**Acceptance Criteria:**
1. Given an ADR in "Draft" status, when an Author adds a note titled "Alternative approaches considered", then the note must be saved with the Author's identity, a timestamp, and the Markdown body.
2. Given a note exists on an ADR, when the Author edits the note body, then the system must save the update and record the modification timestamp.
3. Given an ADR transitions to "Approved", when a user attempts to edit an existing note, then the system must deny the edit.

---

## L2-017: Comments
**Traces to:** L1-007

Users with Author, Reviewer, or Admin roles must be able to add threaded comments to ADRs in any status. Comments must support Markdown formatting.

**Acceptance Criteria:**
1. Given an ADR in "In Review" status, when a Reviewer adds a comment "Consider the impact on latency", then the comment must appear in the ADR's comment thread with the Reviewer's name and timestamp.
2. Given an existing comment on an ADR, when another user replies to it, then the reply must be nested under the parent comment as a thread.
3. Given a user with the "Viewer" role, when they attempt to add a comment, then the system must deny the action with a 403 Forbidden response.
4. Given a comment exists, when the comment author edits it within 15 minutes of creation, then the system must allow the edit and display an "edited" indicator.

---

## L2-018: Architecture Document Attachments
**Traces to:** L1-007

Users must be able to upload architecture documents (PDF, PNG, JPG, SVG, DRAWIO) to ADRs in "Draft" or "In Review" status. Each attachment must have a display name and optional description. The system must enforce a maximum file size of 10 MB per attachment.

**Acceptance Criteria:**
1. Given an ADR in "Draft" status, when an Author uploads a 5 MB PDF named "system-context-diagram.pdf" with description "C4 Level 1 diagram", then the file must be stored and associated with the ADR.
2. Given an Author attempts to upload a 15 MB file, when the upload is validated, then the system must reject it with an error: "File size must not exceed 10 MB."
3. Given an Author attempts to upload a .exe file, when the upload is validated, then the system must reject it with an error: "File type not permitted."
4. Given an ADR has 3 architecture documents attached, when a user views the ADR detail page, then all 3 attachments must be listed with name, file type icon, size, and upload date.

---

## L2-019: Full-Text Search
**Traces to:** L1-008

The system must provide full-text search across ADR titles, Markdown body content, and tags within the user's active tenant. Search results must be ranked by relevance.

**Acceptance Criteria:**
1. Given 50 ADRs exist in the tenant, when a user searches for "event-driven", then ADRs containing "event-driven" in the title or body must be returned, ordered by relevance score.
2. Given a search query "microservices", when results are returned, then each result must display the ADR number, title, status, and a text snippet showing the match in context.
3. Given a user in tenant A searches for content that exists only in tenant B, when results are returned, then zero results must be shown.

---

## L2-020: ADR Filtering and Browsing
**Traces to:** L1-008

The system must allow users to filter the ADR list by status, author, date range, and tags. The default view must show all ADRs sorted by last-modified date descending.

**Acceptance Criteria:**
1. Given 20 ADRs exist with mixed statuses, when a user filters by status "Draft", then only ADRs in Draft status must be displayed.
2. Given a user applies filters for status "Approved" and author "Alice", when results are displayed, then only ADRs matching both criteria must appear.
3. Given no filters are applied, when the ADR list loads, then all ADRs must be displayed sorted by last-modified date descending.
4. Given a user applies a tag filter "security", when results are displayed, then only ADRs tagged with "security" must appear.

---

## L2-021: ADR Tagging
**Traces to:** L1-008

Authors and Admins must be able to assign tags to ADRs. Tags must be scoped to the tenant and reusable across ADRs. The system must suggest existing tags as the user types.

**Acceptance Criteria:**
1. Given an Author editing ADR-010, when they type "sec" in the tag input, then the system must suggest existing tags starting with "sec" (e.g., "security", "secrets-management").
2. Given an Author adds a new tag "observability" that does not yet exist, when the ADR is saved, then the tag must be created within the tenant's tag pool and applied to the ADR.
3. Given a tag "deprecated-pattern" exists in tenant A, when a user in tenant B types "deprecated", then the tag must not appear in suggestions.

---

## L2-022: Audit Trail for ADR Changes
**Traces to:** L1-009

The system must record an immutable audit entry for every ADR content change, status transition, approver assignment, approval decision, and artifact modification. Each entry must include the actor, action, timestamp, and a summary of the change.

**Acceptance Criteria:**
1. Given a user edits ADR-005's body content, when the edit is saved, then an audit entry must be created with the user's ID, action "content_updated", timestamp, and a diff summary.
2. Given ADR-005 transitions from "Draft" to "In Review", when the transition completes, then an audit entry must be created with action "status_changed", previous status, new status, and the actor.
3. Given a tenant admin queries the audit trail for ADR-005, when results are returned, then all entries must be in reverse chronological order and must not be editable or deletable by any user.

---

## L2-023: ADR Version History
**Traces to:** L1-009

The system must maintain a version history for each ADR's Markdown content. Users must be able to view any previous version and compare two versions side by side.

**Acceptance Criteria:**
1. Given ADR-005 has been edited 5 times, when a user views the version history, then all 5 versions must be listed with version number, author, and timestamp.
2. Given a user selects version 2 and version 4 for comparison, when the diff view loads, then additions must be highlighted in green and deletions in red.
3. Given a user views version 3, when the content renders, then it must display the exact Markdown content as it existed at version 3.

---

## L2-024: Input Validation and Sanitization
**Traces to:** L1-010

All user input must be validated on both client and server. Markdown content must be sanitized on render to prevent XSS. API inputs must be validated against defined schemas.

**Acceptance Criteria:**
1. Given a user submits an ADR title containing `<script>alert('xss')</script>`, when the title is rendered anywhere in the UI, then the script must not execute and must be displayed as escaped text.
2. Given a user submits Markdown containing `![img](javascript:alert(1))`, when the preview renders, then the malicious link must be stripped.
3. Given an API request with a body exceeding 1 MB, when the server processes it, then the request must be rejected with a 413 Payload Too Large response.
4. Given an API request with an ADR title exceeding 200 characters, when the server validates it, then the request must be rejected with a 400 Bad Request response indicating the constraint.

---

## L2-025: HTTPS and Token Security
**Traces to:** L1-010

All communications must be encrypted via TLS 1.2 or higher. JWT tokens must be stored securely (HttpOnly cookies or secure storage) and must not be exposed to JavaScript.

**Acceptance Criteria:**
1. Given a client attempts to connect via HTTP, when the request reaches the server, then it must be redirected to HTTPS with a 301 response.
2. Given a user authenticates and receives a JWT, when the token is stored, then it must be in an HttpOnly, Secure, SameSite=Strict cookie.
3. Given a JWT has expired, when the client makes an API request, then the server must return a 401 Unauthorized response.

---

## L2-026: Cross-Tenant Security Enforcement
**Traces to:** L1-010

The system must prevent all forms of cross-tenant data access including direct object reference attacks, tenant ID manipulation in API requests, and SQL injection targeting tenant isolation.

**Acceptance Criteria:**
1. Given user in tenant A modifies the tenant ID in an API request header to tenant B's ID, when the request is processed, then the system must reject it or resolve the tenant from the authenticated session (not the request header).
2. Given an API endpoint accepts a resource ID, when the resource belongs to a different tenant, then the system must return 404.
3. Given a search query containing SQL injection payload `' OR tenant_id = 'other-tenant`, when the query is executed, then the system must parameterize the query and return no cross-tenant results.

---

## L2-027: API Response Time
**Traces to:** L1-011

API endpoints must respond within 200ms at the 95th percentile for list operations (up to 50 items) and within 100ms at the 95th percentile for single-resource operations under normal load.

**Acceptance Criteria:**
1. Given 500 ADRs exist in a tenant, when a user requests the ADR list with default pagination (50 items), then the server must respond within 200ms at p95.
2. Given a single ADR with 10 attachments and 20 comments, when a user requests the ADR detail, then the server must respond within 100ms at p95.
3. Given 50 concurrent users performing mixed read/write operations, when load is sustained for 5 minutes, then p95 latency must not exceed 500ms for any endpoint.

---

## L2-028: Pagination
**Traces to:** L1-011

All list endpoints must support cursor-based pagination with a configurable page size (default 25, maximum 100). The UI must implement infinite scroll or paginated navigation.

**Acceptance Criteria:**
1. Given 200 ADRs exist, when a user requests the first page with page size 25, then exactly 25 ADRs must be returned along with a cursor for the next page.
2. Given a user requests page size 150, when the request is validated, then the system must cap the page size at 100 and return 100 items.
3. Given a user scrolls to the bottom of the ADR list, when more items exist, then the next page must load automatically (infinite scroll) or a "Load more" button must be visible.

---

## L2-029: Responsive Layout - ADR List
**Traces to:** L1-012

The ADR list view must adapt to all viewport sizes. On extra-small viewports (< 576px), the list must display as cards with essential information. On medium and larger viewports (>= 768px), the list must display as a table with sortable columns.

**Acceptance Criteria:**
1. Given viewport width < 576px, when the user views the ADR list, then each ADR must display as a card showing the ADR number, title, status badge, and last-modified date in a single-column stacked layout.
2. Given viewport width >= 768px, when the user views the ADR list, then ADRs must display in a table with columns: Number, Title, Status, Author, Last Modified, and Tags.
3. Given viewport width between 576px and 767px, when the user views the ADR list, then the layout must use a compact card or condensed table format that fits without horizontal scrolling.

---

## L2-030: Responsive Layout - Markdown Editor
**Traces to:** L1-012

The Markdown editor must adapt to viewport size. On viewports >= 768px, the editor must display in split-pane mode (edit + preview). On viewports < 768px, the editor must display in a tabbed mode with toggle between edit and preview.

**Acceptance Criteria:**
1. Given viewport width >= 768px, when the user opens the ADR editor, then the edit pane and live preview pane must display side by side.
2. Given viewport width < 768px, when the user opens the ADR editor, then the editor must display in full width with a toggle button to switch between "Edit" and "Preview" views.
3. Given the user is in "Edit" mode on a mobile viewport, when they tap "Preview", then the editor must switch to render the current Markdown content in preview mode without losing unsaved edits.

---

## L2-031: Responsive Layout - ADR Detail
**Traces to:** L1-012

The ADR detail page must display the ADR content, supporting artifacts (meeting notes, notes, comments, documents), and workflow actions in a responsive layout. On large viewports, artifacts must display in a sidebar. On small viewports, artifacts must be accessible via collapsible sections below the main content.

**Acceptance Criteria:**
1. Given viewport width >= 992px, when the user views an ADR detail page, then the main content must occupy approximately 2/3 of the width with a sidebar showing artifacts, approvers, and workflow actions.
2. Given viewport width < 768px, when the user views an ADR detail page, then the main content must be full width with collapsible accordion sections for "Meeting Notes", "Notes", "Comments", "Documents", and "Approval Status" below the content.
3. Given viewport width < 576px, when the user views workflow action buttons, then they must stack vertically and span the full width.
